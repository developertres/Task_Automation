Cummins utilizes the Microsoft Windows operating system. Windows runs processes in the background, and each process runs in the context of a user. Some of these services run as built-in accounts, such Network Service or Local System, while other services must run as a specific user which is granted access to resource on the network. These specific user accounts must be managed and access granted only as required to ensure that they are both functional and secure.

This document will detail the requirements for these service accounts within the Cummins environment, will include the request form, and the steps for processing.

Exceptions to the following requirements can be made using the operational exception process. See Appendix B.

Provisioning

Most Windows systems in the Cummins Windows environment are part of an Active Directory domain. The service accounts will be provisioned as an AD domain account.

The only locally created service accounts will be for Windows systems that are not able to be in an Active Directory domain, and must be setup in a Workgroup.

Naming

Windows accounts are limited to 20 characters. A standard naming structure is needed to easily differentiate service accounts from normal users, identifies what the account's role is, and stays within the 20 character limit. The following components will make up a standard service account name:

  • A prefix, in uppercase, of the domain name
  • An underscore character to separate domain from the role
  • A short indicator of the application
  • An indicator of the service role

An example service account would be CED-vSphereSQL, and this would be in the CED domain, and would be the SQL service for the vSphere application.

Organizational Unit

Service accounts will all reside within the same Organizational Unit (OU) in Active Directory.

The OU will be named “Service Accounts” and will be located in the top level “Admin” OU.

The operations team managing service accounts will be delegated full control over User objects in the “Service Accounts” OU.

Passwords

Passwords will be set by the operations team managing the service accounts. The passwords will be created using a random password generator with sufficient length (at least 12 characters) and complexity.

The passwords will be stored in a centralized secure password database. Only individuals on the operations team or the application team (listed on the service account form) can have access to the password.

Passwords will be setup as non-expiring, and set so as the user cannot change the password.

Privileges

Service accounts can be delegated administration rights over specific servers. This right is given by adding the service account to a security group called g_ServerName_Admin (where ServerName is the name of the server) within the Active Directory domain.

Service accounts will require the ability to login to a server to execute tasks.

Remote Desktop

A service account will not have the capability to remote desktop into a server. It will have local login rights only. Applications should not normally require being run in remote desktop mode.

If a user must run an install or execute tasks as a service account in remote desktop mode, they use the Run As feature within Windows to run the task as the service account.

Internet Access

Access to the internet will be limited for the service accounts. The primary purpose is to execute tasks and run processes. It is known that some services require access to access the internet for application updates, and this can be granted via operational exceptions.

This will be implemented by adding the service account to the g_NoInternet security group.

UID/GID Attribute

Linux/Unix systems use a local user ID number (UID) and group ID number (GID) to identify users on the system. These UID: GID numbers are a simple integer, for example 501:501. Microsoft Windows Active Directory use a different user ID structure to identify users, groups, and machines. Each ID, also called a Security Identifier (SID) is constructed of different segments that identify the security version, the issuing authority type, the machine, and the identity itself. In addition it’s not a part of Kerberos authentication and Microsoft has extended their schema/added an attribute in Active Directory for environments where in Linux/Unix OS flavors are also used.

Account Listing

The service accounts will be exported to a file for review on a weekly basis. This file will be stored on network based storage that is easily accessible. The weekly exports will be kept long term to allow historical views of service accounts

Account Permission Listing

The groups that the service accounts will be exported to a file for review on a weekly basis. This file will be stored on network based storage that is easily accessible. The weekly exports will be kept long term to allow historical views of groups that provide permissions to service accounts

Account Permission Review

The groups that the service accounts will be reviewed on a quarterly basis. The basis of this review is to identify any groups that the service accounts are a member of that are not authorized, or should no longer be required.

Owners of the service accounts will be contacted for questions.

Account Activity Review & Disabling

Tools will be used to analyze service accounts on a quarterly basis. The tool will identify the last time the service account authenticated with the domain. Any service account that has not authenticated in the last 60 days will be disabled, and will be deleted after being disabled for over 30 days.

Account Owner Review

The owners that the service accounts will be reviewed on a quarterly basis. Any WWID’s that are invalid will be addressed to the listed Supervisor.

Accounts that cannot have a valid owner will be disabled until an owner can be identified.
